# SDLC mode ‚Äî design spec

Date: 2026-07-16
Status: approved for build
Parent spec: `2026-07-15-full-auto-board-runner.md` (SDLC was its documented future variant).

## Intent

FULL_AUTO has one human gate: story readiness. SDLC mode adds two more, placed where
software teams actually want them: **after planning** and **after review**. A story with
`mode: sdlc` flows gate ‚Üí plan ‚Üí *human approval* ‚Üí build ‚Üí review ‚Üí *human approval* ‚Üí
ship. Every stage is executed by a worker or the runner; every human gate is an
**approval annotation** on the board card. Fail closed everywhere: no approval, no
progress; a rejection blocks the chain with the feedback recorded.

The architecture boundary is unchanged: the server stays storage + lifecycle +
broadcast; the runner owns all orchestration (including reacting to approvals); SQLite
is the source of truth; SSE is a best-effort wake hint backed by reconciliation polls.

## Vocabulary (additions)

- **Stage task** ‚Äî one lifecycle task per SDLC stage, all under the same story spec:
  lanes `sdlc:plan`, `auto` (build ‚Äî reused as-is), `sdlc:review`.
- **Approval annotation** ‚Äî kind `approval`, dataJson
  `{"decision": "approve" | "reject", "stage": "plan" | "review", "feedback": "..."}`.
  Posted by a human from the card UI. The runner is the only consumer.
- **Awaiting approval** ‚Äî a stage task that is `done` whose successor has not been
  enqueued because no `approve` annotation exists yet. A UI state, not a new status.

## Pipeline (SDLC)

1. **Intake** ‚Äî same story form; the `sdlc` mode radio becomes enabled. Frontmatter
   `mode: sdlc` is recorded in the frozen spec.
2. **Gate** ‚Äî same deterministic checks. On pass, `mode` (threaded out of
   `GateEvaluator.GateResult`, which currently discards it) selects the successor:
   `full_auto` ‚Üí lane `auto` (unchanged); `sdlc` ‚Üí lane `sdlc:plan`. The
   existing open/in-progress dedupe helper is parameterized by lane.
3. **Plan** (`sdlc:plan`) ‚Äî the runner claims and spawns a worker (same engine
   machinery, same tmux/worktree naming) with a **plan-only goal prompt**: explore the
   repo read-only, produce an implementation plan (approach, files to touch, risks,
   verify plan) ‚Äî *make no commits*. The worker posts the plan via
   `scripts/runner/report.sh <taskId> plan "<plan markdown>"` (new verb ‚Üí annotation
   kind `plan`), then reports done. The runner completes the task with a handoff
   summarizing the plan. Card sits in Done, **Awaiting approval**.
4. **Approval #1** ‚Äî human reads the plan on the card, clicks **Approve** or
   **Reject** (with feedback). Approve ‚Üí runner enqueues the build task (lane `auto`,
   same spec, same priority). Reject ‚Üí runner marks the stage chain blocked: it
   annotates the plan task and enqueues nothing; `blockedReason` semantics are carried
   on a `progress` annotation (the plan task is already `done`; nothing else moves).
5. **Build** (`auto`) ‚Äî identical to FULL_AUTO execution **except ship is deferred**:
   for a `mode: sdlc` story the runner stops after verify-green + commit, preserves
   the worktree and branch, annotates branch/worktree (dataJson), completes the task,
   and enqueues `sdlc:review`. No push, no PR, no merge yet.
6. **Review** (`sdlc:review`) ‚Äî the runner spawns a review worker in the *same
   worktree* with an adversarial-review goal prompt: read the diff against the default
   branch, check the acceptance criteria and the approved plan, run verify, post
   findings via `report.sh <taskId> review "<findings markdown>"` (annotation kind
   `review`), then report done. Review is advisory text for the human; it makes no
   changes. Task completes; card sits Awaiting approval.
7. **Approval #2** ‚Äî human reads the review. Approve ‚Üí the runner ships directly (no
   worker): `ShipExecutor` against the preserved worktree ‚Äî push/PR/merge under the
   exact same fail-closed repo-config gates as FULL_AUTO ‚Äî then prunes the worktree
   only when merged+clean and posts the final handoff annotations on the review task.
   Reject ‚Üí chain blocked with feedback, worktree preserved for inspection.

## Approval transport (runner-owned, crash-safe)

- **Wake hint**: the runner's existing SSE `task.note` handler adds a branch for
  `annotation.kind == "approval"`. On an approve for a task it recognizes as an SDLC
  stage (lane `sdlc:plan` ‚Üí enqueue `auto`; lane `sdlc:review` ‚Üí ship), it acts
  immediately.
- **Reconciliation (source of truth)**: SSE frames can be missed. On startup and on a
  slow poll (default every 60s), the runner scans tasks in lanes `sdlc:plan` and
  `sdlc:review` with status `done`, reads each task's events
  (`GET /api/tasks/{id}/events`), and applies the newest `approval` annotation that
  has not been acted on. "Acted on" is determined structurally, not by memory:
  - plan approved ‚Üí successor `auto` task for the same spec already exists
    (open/in_progress/done) ‚Üí nothing to do;
  - review approved ‚Üí a `progress` annotation kind data `{"sdlc":"shipped"}` (or the
    ship-result annotation itself) already exists on the review task ‚Üí nothing to do.
  - rejections are acted on by posting the rejection-recorded annotation once
    (dedupe by scanning for it).
- Approvals are idempotent by construction; double-clicks and replayed frames are
  harmless.

## Server additions (storage only)

1. `AnnotationKind`: add `plan`, `review`, `approval`. No other validation changes ‚Äî
   annotations remain append-only facts, valid on done tasks (already true).
2. Nothing else. No new tables, no new endpoints: stage chaining is spec-scoped task
   enqueueing the runner already does; approvals ride the existing annotations API and
   the existing `task.note` SSE frame.

## Runner additions

- `RunnerDaemon`: claim blocks for `sdlc:plan` and `sdlc:review`, dispatched to the
  worker pool under the same concurrency cap as `auto`; approval branch in
  `onSseFrame`; reconciliation poll.
- `SdlcPlanCycle` / `SdlcReviewCycle` (mirroring the gate/auto cycle seam): thin
  orchestrations that reuse `RunExecutor`'s worktree/tmux/engine/report machinery with
  stage-specific goal prompts and completion handling. Prefer extracting the reusable
  run scaffolding over duplicating it; do not fork `RunExecutor` wholesale.
- `RunExecutor` (build stage): learns `mode` (from the spec frontmatter via the same
  parser) ‚Äî for `sdlc`, skip ship, preserve worktree, annotate
  `{"branch": ..., "worktree": ...}`, enqueue `sdlc:review` on completion.
- `report.sh`: new verbs `plan` and `review` posting the matching annotation kinds
  (same POST endpoint, kind differs). `done`/`blocked` unchanged.
- Goal prompts: plan prompt forbids commits (guardrails: read-only exploration, no
  file mutations outside the plan text); review prompt forbids code changes and
  instructs verify + diff reading. Both keep the standard identity/fail-closed
  guardrails verbatim.
- Fake engine: `scripts/runner/fake-worker.sh` gains stage awareness via env
  `SBA_STAGE` (`plan` | `build` | `review`; absent = build, current behavior): plan
  emits a canned plan annotation + done (no commit); review emits a canned review
  annotation + done (no commit); build unchanged. The runner sets `SBA_STAGE` for all
  engines (codex prompt content already differs per stage; env is for the fake).
- Crash recovery: existing reconcile (claimed tasks with dead tmux ‚Üí open) applies to
  the new lanes unchanged; the approval reconciliation poll covers the approval seam.

## Frontend additions

1. **StoryForm**: enable the `sdlc` radio (remove `disabled`, drop "coming soon").
2. **Card detail**: render `plan` and `review` annotations as full-width preformatted
   blocks (they carry markdown-ish text, distinct kind badges). When a task is
   `done` in lane `sdlc:plan`/`sdlc:review` and no decision has landed, show an
   **Awaiting approval** chip and an approval box: Approve button, Reject button +
   required feedback textarea (mirrors `SteerBox` ‚Äî posts the `approval` annotation
   with dataJson). After a decision annotation exists, the box collapses to a
   decision record ("Approved by nathan at ‚Ä¶" / rejection + feedback).
3. **Board**: no column changes (lanes already auto-discovered in the lane filter).
   Card chips: stage chip for sdlc lanes (`plan` / `review`), awaiting-approval state
   visible on the card face.
4. Annotation kind label/style maps updated for `plan`, `review`, `approval`.

## Non-goals

- Revise-and-resubmit on rejection (rejection blocks; resubmit = new story ‚Äî the
  blocked-card prefill is a separate stretch item).
- Per-repo approval policies in runner config; multiple approvers; approval auth.
- MCP parity for approvals (REST + UI only, like annotations).
- Changing FULL_AUTO behavior in any way ‚Äî `mode: full_auto` paths must be
  byte-identical in behavior.

## Slices

1. Server: `AnnotationKind` `plan`/`review`/`approval` (+ tests).
2. Runner: gate mode-branch, plan/review cycles, build-stage ship deferral,
   approval SSE + reconciliation, ship-on-approval, `report.sh` verbs, fake-worker
   stages (+ tests).
3. Frontend: sdlc radio, approval box, awaiting-approval states, kind rendering
   (+ tests).
4. E2E + docs (`docs/architecture.md` SDLC section, README, this spec linked).
